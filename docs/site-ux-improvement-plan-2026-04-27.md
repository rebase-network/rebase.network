# Rebase.network UI/UX 问题修复方案与改版执行计划

日期：2026-04-27

依据：`docs/site-ux-review-2026-04-27.md`

目标：把真实页面审阅中发现的问题转成可排期、可实现、可验收的执行方案。本文不替代设计规范，也不直接要求一次性重做全站；它用于指导后续拆 PR、排优先级和验收。

## 总体判断

当前更合适的流程不是直接进入大范围页面重构，而是先做“信任与可用性修复”，再做“信息架构和视觉密度优化”。

推荐流程：

1. 先修会误导用户、破坏信任或阻断浏览的问题。
2. 再修内容结构、页面层级、详情页效率问题。
3. 最后做栏目页文案、贡献者页、历史归档等较重的体验优化。

这样可以避免一次性改动过大，也方便把产品判断、内容运营和工程实现分开评审。

## 执行原则

- 以真实页面问题为准，不因为文档里已有描述就跳过线上问题。
- 不做全站视觉推倒重来，优先保留现有暖色、编辑感、社区媒体风格。
- 每个 PR 只解决一个清晰目标，避免同时改数据、组件和大段文案。
- 能通过内容或数据清洗解决的问题，不优先用复杂 UI 包装。
- 所有可点击暗示必须有真实行为；所有不可用操作必须有明确状态。
- 列表页优先可扫读，详情页优先让正文靠前，移动端优先减少重复信息。

## 优先级分批

### Batch 1：信任与基础可用性

目标：先修复会误导用户、阻断浏览、影响订阅和 SEO 的问题。

建议包含：

- UX-01 活动状态与 CTA 规则
- UX-02 GeekDaily 分页可访问性
- UX-03 RSS 与 sitemap 可用性复核
- UX-04 招聘数据质量与投递 CTA
- UX-05 文章摘要截断
- UX-15 Footer 社交链接可访问名称

### Batch 2：内容结构与阅读效率

目标：减少重复信息，提高详情页和文章阅读质量。

建议包含：

- UX-06 文章正文标题层级
- UX-07 详情页元信息去重
- UX-08 About 页面内容重组
- UX-12 文章列表信息密度
- UX-13 GeekDaily 顶部“推荐内容”入口处理

### Batch 3：栏目页与归档体验优化

目标：优化较大范围的信息架构和视觉密度，提升社区媒体浏览体验。

建议包含：

- UX-09 贡献者页参与路径重组
- UX-10 贡献者去重与头像策略
- UX-11 栏目页长标题收敛
- UX-14 活动历史归档组织

## 详细方案

### UX-01 活动状态与 CTA 规则

影响页面：`/events`、`/events/{slug}`、首页活动卡片如有复用。

现状问题：已结束活动仍显示“立即报名”，与“往期活动”状态冲突。

修改目标：让用户一眼知道活动是否还能参与，并且不会点击错误的报名操作。

推荐方案：

- 活动状态按时间和报名配置生成：`upcoming`、`open`、`closed`、`past`。
- 只有未结束且存在有效报名链接的活动显示“立即报名”。
- 已结束活动统一显示“查看详情”或“查看回顾”。
- 如果历史活动保留原始报名链接，文案改为“历史活动链接”，弱化为次要操作。
- 活动详情页只保留一个主 CTA，避免正文、侧栏和底部重复出现同一个报名入口。

验收标准：

- 过去日期活动不再出现“立即报名”。
- 即将举办且有报名链接的活动显示“立即报名”。
- 没有报名链接的活动不显示空 CTA。
- 列表页和详情页状态文案一致。
- 移动端卡片中状态和 CTA 不换行拥挤。

建议 PR：`fix(events): align event status actions`

### UX-02 GeekDaily 分页可访问性

影响页面：`/geekdaily`

现状问题：页面有“上一页 / 第 1 页 / 下一页”，但抽取结果中分页疑似不是可访问链接。

修改目标：极客日报历史可以稳定浏览、复制 URL、被搜索引擎和辅助技术访问。

推荐方案：

- 使用真实链接实现分页，例如 `/geekdaily?page=2`。
- 第一页的“上一页”使用 disabled 状态，不渲染为可点击链接。
- 最后一页的“下一页”使用 disabled 状态。
- 搜索、年份、标签、页码参数应能组合：`/geekdaily?q=agent&year=2026&page=2`。
- 搜索或筛选条件变化时回到第一页。
- 保留服务端渲染的基础结果，前端增强只能作为增强，不应是唯一浏览路径。

验收标准：

- 不依赖 JS 时，点击“下一页”可以进入第 2 页。
- 浏览器地址栏能体现当前筛选和分页状态。
- 搜索结果为空时显示明确空状态和“清空筛选”入口。
- 键盘 Tab 可以聚焦分页链接。
- 屏幕阅读器可读出“上一页”“下一页”和当前页。

建议 PR：`fix(geekdaily): make archive pagination accessible`

### UX-03 RSS 与 sitemap 可用性复核

影响页面：`/rss.xml`、`/geekdaily/rss.xml`、`/articles/rss.xml`、`/events/rss.xml`、`/who-is-hiring/rss.xml`、`/sitemap.xml`、`/robots.txt`

现状问题：RSS 和 sitemap 是 V1 范围，但审阅时存在访问失败、超时或自动化工具无法读取的风险。

修改目标：订阅器、搜索引擎、监控和命令行工具都能稳定读取 XML 输出。

推荐方案：

- 在 CI 或部署后检查中加入 RSS 和 sitemap smoke check。
- 对每个 feed 检查状态码、`Content-Type`、XML 可解析性、item 数量和 item link。
- sitemap 如果条目过多导致响应慢，改为 sitemap index 或缓存静态结果。
- `robots.txt` 中的 sitemap 链接必须与生产域名一致。
- 明确 HTTP 到 HTTPS 的行为，推荐 301 跳转到 HTTPS。

验收标准：

- `curl -Iv https://rebase.network/rss.xml` 返回 200。
- `curl -Iv https://rebase.network/sitemap.xml` 返回 200，响应时间在可接受范围内。
- RSS 阅读器可以订阅站点总 feed 和频道 feed。
- sitemap XML 可被解析，包含静态页面和内容详情页。
- `http://rebase.network/` 不应空响应，应 301 到 HTTPS 或返回可解释状态。

建议 PR：`test(web): add rss and sitemap smoke checks`

### UX-04 招聘数据质量与投递 CTA

影响页面：`/who-is-hiring`、`/who-is-hiring/{slug}`

现状问题：招聘公司名可能显示为 `/`；详情页使用“报名表单”等不适合招聘语境的通用提示。

修改目标：招聘列表可信、投递路径明确，不让用户误判入口含义。

推荐方案：

- 发布前校验公司名，空值、`/`、`-` 等占位值不应直接展示。
- 缺失公司名时显示“公司未公开”，并在视觉上弱化。
- 区分三种入口：直接投递链接、联系方式、原始招聘帖。
- CTA 按入口类型生成：
  - 有投递链接：`立即投递`
  - 只有联系方式：`联系招聘方`
  - 只有来源链接：`查看原始招聘信息`
- 详情页去掉“报名表单”措辞，改为招聘语境。
- 列表卡片保留最有决策价值的信息：职位、公司、地点、远程/现场、薪资或范围、发布时间。

验收标准：

- 列表和详情页不出现公司名 `/`。
- 每个公开职位至少有一种可理解的联系或来源方式。
- CTA 文案与真实数据类型一致。
- 无投递链接时不显示“立即投递”。
- 移动端长职位标题和公司名换行后仍可读。

建议 PR：`fix(hiring): clarify job company and apply actions`

### UX-05 文章摘要截断

影响页面：`/`、`/articles`、`/articles/{slug}`

现状问题：文章摘要出现未闭合引号和硬截断，首屏观感像内容未完成。

修改目标：摘要自然、完整、可用于卡片点击决策。

推荐方案：

- 优先使用编辑维护的 `summary` 字段，不从正文机械截断。
- 如果必须自动生成摘要，按句号、问号、感叹号、中文逗号等边界截断。
- 自动截断后补省略号，并避免截断在引号、括号、Markdown 标记中间。
- 后台或构建阶段检查摘要长度和未闭合符号。

验收标准：

- 首页和文章列表摘要不出现明显半句话。
- 摘要长度在桌面和移动端卡片中都可控。
- 文章详情页摘要与列表摘要一致或有明确规则。
- 无摘要文章有合理 fallback，不暴露 Markdown 残片。

建议 PR：`fix(articles): improve summary fallback`

### UX-06 文章正文标题层级

影响页面：`/articles/{slug}`

现状问题：页面已有 H1，正文 Markdown 仍出现多个 H1。

修改目标：详情页语义结构稳定，正文层级从 H2 开始。

推荐方案：

- 渲染文章正文时，将正文内 H1 自动降级为 H2。
- 或在发布前 lint Markdown，阻止正文包含 H1。
- 编辑器中增加提示：文章标题由标题字段管理，正文从二级标题开始。

验收标准：

- 每个文章详情页只有一个 H1。
- 正文一级段落视觉层级低于页面标题。
- Markdown 中存在 `#` 时，线上渲染不破坏页面语义。

建议 PR：`fix(articles): normalize body heading levels`

### UX-07 详情页元信息去重

影响页面：`/articles/{slug}`、`/events/{slug}`、`/who-is-hiring/{slug}`、`/geekdaily/{slug}`

现状问题：日期、作者、状态、地点、报名入口等信息在 hero、侧栏和正文附近重复出现，正文被推远。

修改目标：详情页让用户更快进入正文，同时保留必要判断信息。

推荐方案：

- Hero 只保留最核心信息：
  - 文章：标题、摘要、发布日期、作者。
  - 活动：标题、时间、城市/地点、状态、主 CTA。
  - 招聘：职位、公司、地点/远程、薪资、主 CTA。
  - GeekDaily：期号、日期、摘要或推荐数量。
- 侧栏只展示补充信息，不重复 hero 已出现的内容。
- 底部相关内容保持轻量，不再重复长说明。
- 移动端优先排序：标题、关键元信息、主 CTA、正文、补充信息。

验收标准：

- 同一详情页中同一元信息不在首屏附近重复出现。
- 移动端首屏可以看到正文开始或明确的正文入口。
- 活动详情页报名链接不重复出现三次以上。
- 去重后不丢失必要信息。

建议 PR：`refactor(web): reduce detail page metadata repetition`

### UX-08 About 页面内容重组

影响页面：`/about`

现状问题：About 页面与首页定位文案重复，没有提供足够的第二层信息。

修改目标：About 页从“重复口号”转为解释社区历史、运行方式、参与方式和支持方式。

推荐方案：

- 首屏保留一句清晰定义：Rebase 是什么。
- 下方改为结构化内容：
  - `我们是谁`：社区背景和非营利属性。
  - `我们如何运作`：志愿者、活动、内容、开源协作。
  - `你可以如何参与`：加入社群、贡献内容、参与活动、支持捐赠。
  - `联系方式与社区入口`：社交渠道和协作入口。
- 避免重复首页已有的长定位句。

验收标准：

- About 页首屏不再大段重复首页文案。
- 用户能在 30 秒内找到参与方式和联系方式。
- 页面仍保持社区温度，不变成组织章程式说明。

建议 PR：`docs(content): revise about page structure`

### UX-09 贡献者页参与路径重组

影响页面：`/contributors`

现状问题：开头段落同时解释多个问题，阅读负担重。

修改目标：让读者快速理解贡献者做什么，以及自己如何加入。

推荐方案：

- 将首段拆成三个短模块：
  - `贡献者做什么`
  - `需要什么能力`
  - `如何加入`
- 将“加入我们”或联系入口前置到首屏附近。
- 减少抽象描述，改为具体贡献类型：内容整理、活动协助、开发维护、社群运营。

验收标准：

- 首屏能看到明确参与入口。
- 三个模块每个不超过 2-3 行说明。
- 移动端模块不形成过长竖向堆叠。

建议 PR：`refactor(contributors): clarify contributor onboarding`

### UX-10 贡献者去重与头像策略

影响页面：`/contributors`

现状问题：同一成员在多个分组重复出现，历史贡献者大量使用单字母占位头像。

修改目标：贡献者页更像“真实社区成员展示”，而不是重复数据表。

推荐方案：

- 同一贡献者只展示一次。
- 多角色通过 badge 展示，例如 `活动`、`内容`、`开发`、`运营`。
- 当前贡献者和历史贡献者分区保留，但历史贡献者采用更紧凑布局。
- 缺头像时使用统一默认头像，不使用大面积随机单字母作为主要视觉。
- 历史贡献者数量较多时，使用折叠、分页或按首字母分组。

验收标准：

- 同名贡献者不在同一页面重复出现多张卡片。
- 无头像成员视觉统一，不显得像错误数据。
- 历史贡献者区滚动长度明显缩短或更易扫描。
- 角色信息不丢失。

建议 PR：`refactor(contributors): dedupe people and improve fallbacks`

### UX-11 栏目页长标题收敛

影响页面：`/articles`、`/events`、`/contributors`

现状问题：多个一级页面 H1 使用长句，移动端会推迟真实内容出现。

修改目标：栏目页先帮助用户确认当前位置，再快速进入内容。

推荐方案：

- H1 使用短栏目名：
  - `社区文章`
  - `社区活动`
  - `贡献者`
- 情绪化或品牌化表达下移到摘要，并控制在 1-2 行。
- 列表页 hero 高度收敛，首屏露出列表内容。

验收标准：

- 移动端 H1 不超过 2 行。
- 桌面首屏可看到列表或核心内容的起始区域。
- 栏目页之间命名风格一致。

建议 PR：`refactor(web): simplify archive page heroes`

### UX-12 文章列表信息密度

影响页面：`/articles`

现状问题：“最新文章”区域缺少摘要，反而不如“全部文章”可判断。

修改目标：最新文章应更容易被理解和点击。

推荐方案：

- 最新文章卡片增加摘要。
- 如果文章数量少，取消“最新文章 / 全部文章”拆分，统一展示一个文章列表。
- 若保留“最新文章”，应让它比普通卡片信息更完整，而不是更少。

验收标准：

- 最新文章区能看到标题、摘要、作者、日期和阅读入口。
- 只有少量文章时页面不显得空。
- 同一篇文章不在上下区域造成明显重复。

建议 PR：`refactor(articles): improve article list hierarchy`

### UX-13 GeekDaily 顶部“推荐内容”入口处理

影响页面：`/geekdaily`

现状问题：“推荐内容 ↗”有箭头，像可点击入口，但实际不可用或不明确。

修改目标：消除错误交互暗示。

推荐方案：

- 如果没有对应页面，去掉箭头，改为普通标签或说明。
- 如果是未来功能，隐藏该入口，避免给用户未上线承诺。
- 如果已有对应资源，补真实链接并给出明确 label。
- 如果代表筛选维度，改为 tab 或 filter chip，不使用外链箭头。

验收标准：

- 页面上带箭头或按钮样式的元素都可点击。
- 不可点击元素不会触发 hover 期待。
- 键盘导航不会聚焦到无行为控件。

建议 PR：`fix(geekdaily): remove misleading recommendation affordance`

### UX-14 活动历史归档组织

影响页面：`/events`

现状问题：历史活动内容从 2021 到 2026 一次性铺开，列表过长，冲淡当前活动。

修改目标：让“当前可参与活动”和“历史活动归档”分层清晰。

推荐方案：

- 首屏优先展示正在报名或即将举办的活动。
- 历史活动默认展示最近 6-8 条。
- 增加年份筛选或年份分组。
- 过去活动卡片压缩展示字段：标题、日期、城市、状态、详情入口。
- 如果活动有回顾文章或视频，历史卡片优先链接“回顾”。

验收标准：

- 用户进入活动页后能先看到当前可参与活动。
- 历史活动可按年份定位。
- 历史卡片不再显示报名型 CTA。
- 页面滚动长度和信息密度更可控。

建议 PR：`refactor(events): organize event archive by status and year`

### UX-15 Footer 社交链接可访问名称

影响页面：全站 footer

现状问题：图标链接没有可读文本或可访问名称。

修改目标：让屏幕阅读器、自动化检查和搜索抽取能识别社交链接目的。

推荐方案：

- 每个社交链接添加 `aria-label`。
- 图标 SVG 设置为装饰性，避免重复读出。
- 外链统一添加安全属性和一致的打开方式。

验收标准：

- 可访问性检查不再报告 footer 图标链接缺少名称。
- 键盘聚焦时能理解当前链接目的。
- 图标视觉不变。

建议 PR：`fix(web): label footer social links`

## 内容与产品待确认事项

在开发前建议确认：

1. 历史活动是否存在“回顾页”“视频链接”“资料链接”等更合适的 CTA。
2. 招聘中公司名缺失时，是否使用“公司未公开”还是隐藏公司字段。
3. GeekDaily 的“推荐内容”是否代表未来页面、筛选器，还是应删除。
4. About 页是否需要补充社区历史、组织方式、捐赠说明等真实内容。
5. 贡献者多角色展示是否以人为主，还是继续以角色分组为主。
6. 活动归档默认展示多少条历史活动。
7. 栏目页 H1 是否统一使用短栏目名。

## QA 与验收清单

每个批次完成后至少验证：

- 桌面宽度：1440px。
- 移动宽度：390px 或 430px。
- 键盘 Tab 顺序。
- 无 JS 基础浏览能力，尤其是 GeekDaily 分页。
- RSS 和 sitemap 的状态码、内容类型、XML 解析。
- 详情页首屏是否出现重复元信息。
- 长标题、缺图、缺公司名、缺报名链接等边界数据。
- 所有新文案在中文语境下自然，不混用“报名”和“投递”。

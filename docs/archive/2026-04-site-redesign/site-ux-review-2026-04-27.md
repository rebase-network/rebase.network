# Rebase.network 真实页面产品与 UI/UX 审阅

审阅日期：2026-04-27

审阅目标：先阅读仓库文档理解 V1 公共站点目标，再只以线上真实页面 `https://rebase.network/` 的实际呈现作为问题判断依据。未修改现有代码。

## 目标理解

Rebase V1 公共站点应是内容优先的社区媒体站点，而不是营销官网。核心任务是快速说明 Rebase 是什么，持续呈现社区正在发生的事，让读者能高效浏览极客日报、招聘、文章、活动、贡献者信息，并通过 RSS 订阅长期内容。

审阅时重点关注：

- 内容是否清晰、可读、可扫读。
- 核心页面和详情页是否完整可访问。
- 卡片、按钮、链接的交互暗示是否与真实行为一致。
- 活动、招聘、文章、极客日报等内容是否存在误导或数据质量问题。
- 视觉表达是否服务于社区媒体阅读，而不是堆叠装饰或长文案。

## 主要问题与建议

### P0/P1 功能与信任问题

#### 1. 历史活动仍显示“立即报名”，会误导用户

页面：`/events`

历史活动列表中，2021、2022、2023、2024、2026 年已结束活动均显示“往期活动”和“立即报名”。例如“Rebase黑客松筹备进展汇报二”“Global HackerHouse：2023 Coding Oasis in China”“Hashkey Hacker House 重磅启动”等历史活动都出现该操作文案。

问题影响：

- 用户会误以为历史活动仍可报名。
- “往期活动”和“立即报名”同时出现，状态与操作冲突。
- 会削弱活动页对时间状态的可信度。

建议：

- 对已结束活动统一使用“查看详情”“查看回顾”或“活动归档”。
- 只有未开始且存在有效报名链接的活动显示“立即报名”。
- 若历史活动保留外部链接，应标注为“原活动链接”或“历史报名页”，避免当作当前报名入口。

#### 2. GeekDaily 列表分页疑似不可点击

页面：`/geekdaily`

页面显示“共 1830 期”“默认展示最近 10 期”，底部出现“上一页 / 第 1 页 / 下一页”，但页面抽取结果中“上一页”和“下一页”没有可访问链接标记。

问题影响：

- 如果“下一页”不是实际链接，用户无法浏览第 2 页及更早期数。
- 即使有前端 JS 绑定，也会影响无 JS、SEO、辅助技术和爬虫访问。
- 这与“极客日报历史可浏览、可搜索”的核心目标冲突。

建议：

- 将分页改为真实 `<a href="/geekdaily?page=2">下一页</a>` 链接。
- 当前不可用的“上一页”应禁用并带明确 disabled 状态。
- 搜索、年份筛选、分页的 URL 参数应可组合并可复制分享。

#### 3. RSS 与 sitemap 的自动化访问存在不稳定风险

页面：`/rss.xml`、`/geekdaily/rss.xml`、`/sitemap.xml`

站点页面中多处提供 RSS 入口，`robots.txt` 也指向 `https://rebase.network/sitemap.xml`。但审阅工具打开 `/rss.xml`、`/geekdaily/rss.xml` 返回 “Failed to fetch ... (400) OK”，打开 `/sitemap.xml` 超时。本地命令行访问生产域名也出现 TLS 连接关闭现象，但普通页面可通过浏览器访问。

问题影响：

- RSS 和 sitemap 是 V1 明确范围，出错会直接影响订阅、SEO、监控与自动化测试。
- 如果只在浏览器页面可见、订阅器或爬虫不可用，会造成“有入口但不可用”的体验落差。

建议：

- 从多网络环境执行 `curl -Iv https://rebase.network/rss.xml`、`curl -Iv https://rebase.network/sitemap.xml` 和 RSS 阅读器订阅测试。
- 确认 XML 响应状态为 200，`Content-Type` 为 RSS/XML 相关类型。
- 为 RSS 和 sitemap 增加部署后监控，检查状态码、响应时间、XML 是否可解析。
- sitemap 如因条目过多导致超时，应拆分 sitemap index 或缓存静态输出。

#### 4. 招聘信息存在明显数据质量问题

页面：`/who-is-hiring`

招聘列表中有条目公司名显示为 `/`，例如“Node.js全栈开发工程师”对应公司字段为 `/`。招聘详情页样本中也存在“请通过下方报名表单或公开链接投递”的通用提示，但实际投递方式可能是微信、Telegram、邮箱或 GitHub issue。

问题影响：

- 公司字段为 `/` 会显得数据未清洗，降低招聘频道可信度。
- “报名表单”对招聘语境不准确，且 GitHub 原帖不一定等同于投递入口。
- 求职者无法快速判断是直接投递、联系招聘方，还是查看原始 issue。

建议：

- 发布前校验公司名，缺失时不允许发布，或显示“公司未公开”并降低视觉权重。
- 区分 `apply_url`、联系方式、原始 GitHub issue 三类入口。
- CTA 文案按实际数据生成：例如“立即投递”“联系招聘方”“查看原始招聘信息”。

### P1 信息架构与内容问题

#### 5. 首页和文章卡片摘要存在截断错误

页面：`/`、`/articles/6-rebuild-rebase-network`

首页文章摘要和文章详情页摘要中，“这也是我们为什么叫 ‘Rebase” 缺少闭合引号，呈现为被硬截断的句子。

问题影响：

- 首屏内容显得未完成，影响第一印象。
- 文章卡片摘要承担引导阅读任务，错误截断会降低点击意愿。

建议：

- 摘要字段应由编辑维护，不建议直接从正文硬截断。
- 如必须自动截断，应按中文标点、英文引号和最大长度做安全截断，并补省略号。

#### 6. 文章详情正文出现多个一级标题，层级不稳定

页面：`/articles/6-rebuild-rebase-network`

文章详情已有页面 H1“重建 Rebase 官网和相关运营工具”，正文中又出现“# 写在前面”“# 正在实施的计划”“# 下一步”等一级标题。

问题影响：

- 页面语义结构混乱，不利于 SEO、目录生成和辅助技术阅读。
- 视觉上多个一级标题会抢夺主标题层级，让长文阅读节奏不稳定。

建议：

- 文章正文中的 Markdown 顶级标题统一下调为 H2。
- 在后台发布前增加 Markdown heading lint，避免正文出现 H1。

#### 7. 详情页元信息重复，正文被推远

页面：`/articles/6-rebuild-rebase-network`、`/events/18-rebase-hacker-house-wuhan`、`/who-is-hiring/15-java`

文章详情中日期、阅读时长、作者在 hero 和“文章信息”中重复。活动详情中时间、地点、状态在 hero、正文旁信息区和“活动信息”中重复，报名链接也至少出现三次。招聘详情中岗位摘要、正文和“投递方式/继续关注”结构也有重复感。

问题影响：

- 详情页正文的开始位置被推后。
- 用户需要多次读到相同信息，降低内容页效率。
- 移动端会更明显，因为重复信息会占用更多首屏高度。

建议：

- Hero 只保留最关键判断信息，例如标题、状态、时间/地点、主 CTA。
- 侧栏或底部信息区只补充未出现的信息。
- 对活动报名链接保留一个主 CTA，正文中的裸链接可作为补充但不重复强调。

#### 8. About 页面与首页定位文案重复

页面：`/about`

About 页首屏标题和“我们是谁”段落都在重复“由中国开发者们在业余时间用热爱建立的开发者社区”“非营利、公益性”等信息。首页也已经使用了高度相似的定位文案。

问题影响：

- About 页没有提供比首页更深一层的信息。
- 页面显得像定位口号重复，而不是社区历史、组织方式、参与方式的解释。

建议：

- About 首屏保留一句清晰定义即可。
- “我们是谁”改为更具体的社区历史、运行机制、志愿者协作方式。
- 将“社区入口与联系”“捐赠支持”等工具信息设计成更易扫描的结构化区块。

#### 9. 贡献者页开头信息过长，Q&A 混在一个段落中

页面：`/contributors`

页面开头同时解释贡献者做什么、需要什么能力、如何成为贡献者，但这些问题全部挤在一个长段落中。

问题影响：

- 首屏阅读负担重，用户难以快速找到“如何加入”。
- 与贡献者页应表达“真实的人”和“参与路径”的目标不完全匹配。

建议：

- 将“贡献者做什么 / 需要什么能力 / 如何加入”拆成 3 个小模块。
- “加入我们”按钮应在首屏或首个模块中更明确出现。
- 首段减少抽象描述，直接说明贡献方式和参与入口。

#### 10. 贡献者页成员重复和占位头像降低观感

页面：`/contributors`

Frozen、Shooter、Harry 等成员在不同分组中重复出现。历史贡献者中大量成员显示为单字母或单字头像，如 `A`、`C`、`Z`、`f`、`北`、`张`、`昊` 等。

问题影响：

- 重复卡片让用户难以理解这是不同角色还是重复数据。
- 大量字母占位头像让页面从“真实的人”变成“数据表”，削弱社区温度。
- 贡献者页总长度很长，视觉节奏容易疲劳。

建议：

- 同一贡献者可展示一次，并使用多角色 badge 标注参与领域。
- 没有头像时使用统一、克制的默认头像样式，不要让单字母占位成为页面主要视觉。
- 历史贡献者可折叠、分页或按首字母索引展示，减少一次性长列表。

### P2 视觉与体验问题

#### 11. 多个一级页面标题过长，移动端可读性风险高

页面：`/articles`、`/events`、`/contributors`

示例：

- 文章页：“记录最新观点，思想的碰撞总是闪耀的，灵魂的交流总是感动的”
- 活动页：“人与人的联系就藏在交流的那一瞬间，每一场活动都是窥探新世界的机遇”
- 贡献者页：“社区不是抽象概念，它总是由具体的人把内容、活动和协作推着往前走。”

问题影响：

- 标题更像宣传句，不像栏目入口。
- 移动端会形成多行大标题，推迟真实列表内容出现。
- 用户扫导航后进入页面，首先需要的是栏目确认和内容入口，而不是再次阅读长口号。

建议：

- 将 H1 改为短栏目名，如“社区文章”“社区活动”“贡献者”。
- 把情绪化描述放入摘要，并限制在 1-2 行。
- 列表页首屏应更快露出真实内容。

#### 12. 文章列表的“最新文章”信息量不足

页面：`/articles`

“最新文章”区域只展示日期、封面空链接、阅读时长、作者和标题，没有展示摘要；而“全部文章”中的其他文章反而有摘要。

问题影响：

- 最新文章应是最需要被点击的内容，但缺少判断信息。
- 页面结构显得不平衡。

建议：

- 最新文章卡片补充摘要、作者和明确“阅读全文”入口。
- 若文章数量少，可以取消“最新文章 / 全部文章”拆分，统一使用一个高质量列表。

#### 13. GeekDaily 列表顶部“推荐内容 ↗”像按钮但不可用

页面：`/geekdaily`

页面顶部出现“推荐内容 ↗”，但真实可点击链接只有“订阅 RSS”。“推荐内容 ↗”带箭头，视觉上像外链或按钮。

问题影响：

- 用户会尝试点击但没有结果。
- 与“hover/箭头暗示可点击”的基本交互原则冲突。

建议：

- 如果功能未上线，去掉箭头并改成普通说明文字。
- 如果是入口，应补真实链接。
- 如果是筛选器，应改成清晰的 tab 或 filter chip。

#### 14. 活动列表历史内容过长，缺少年份/状态筛选

页面：`/events`

历史活动从 2021 到 2026 一次性铺开，且每条都展示较多摘要、时间、地点和操作文案。

问题影响：

- 用户很难快速区分“近期可参与活动”和“历史归档”。
- 历史内容过长会冲淡唯一正在报名活动的优先级。

建议：

- 默认展示“正在报名/即将举办”和最近若干历史活动。
- 历史活动增加年份筛选或折叠分组。
- 过去活动的卡片可以更紧凑，保留标题、日期、城市、状态即可。

#### 15. Footer 社交图标缺少可读名称

页面：全站 footer

页面抽取结果中社交链接显示为空文本链接，例如 Bilibili、YouTube、Discord、X、Telegram、微信、QQ、GitHub 图标链接没有可读名称。

问题影响：

- 屏幕阅读器和自动化抽取难以识别链接目的。
- 对 SEO 和可访问性不友好。

建议：

- 为每个图标链接添加 `aria-label`，例如 `aria-label="Rebase Bilibili"`。
- 图标 SVG 保持装饰即可，链接本身必须有可访问名称。

## 综合优先级建议

近期优先处理：

1. 修复历史活动“立即报名”的状态/操作冲突。
2. 确认 GeekDaily “下一页”是否真实可点击，并补齐可访问分页。
3. 复核 RSS 和 sitemap 的状态码、响应时间和 XML 可解析性。
4. 清洗招聘公司名、投递方式和 CTA 文案。
5. 修复文章摘要截断和正文标题层级。

中期优化：

1. 收敛一级页面长标题，让栏目页更快进入真实内容。
2. 精简详情页重复元信息，提高移动端首屏效率。
3. 重构贡献者页的参与路径、重复成员和占位头像。
4. 活动历史按年份/状态组织，减少一次性长列表。
5. 补全 footer 社交链接可访问名称。

整体判断：站点已经具备社区媒体的核心页面与真实内容，但目前最影响体验的是状态与操作不一致、归档浏览可用性风险、内容数据未清洗，以及多个页面过度依赖长口号和重复元信息。优先修复这些问题后，网站会更可信、更易扫读，也更符合“内容优先的开发者社区媒体站点”目标。

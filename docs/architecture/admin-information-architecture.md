# 管理后台信息架构

这个文件描述管理工作台的模块划分和运营任务流。

它回答的是：

- 工作人员在 admin 中会看到哪些顶层模块
- 每个模块要解决什么任务
- 哪些界面能力是非可选的

它不重复系统部署、认证实现或完整字段清单。

## 设计目标

admin 的组织方式应围绕工作人员任务，而不是围绕数据库表。

每个模块都应帮助工作人员回答一个真实问题，例如：

- 今天要发布什么
- 哪一期 GeekDaily 正在维护
- 哪些岗位还在线
- 哪些贡献者属于哪个角色

## 主要角色

- 超级管理员
- 内容编辑
- GeekDaily 编辑
- 招聘编辑
- 活动编辑
- 社区编辑
- 审计人员

## 顶层模块

推荐的顶层导航：

- 仪表盘
- GeekDaily
- 招聘
- 文章
- 活动
- 贡献者
- 页面与导航
- 媒体库
- 工作人员与权限
- 审计日志

## 模块职责

### 仪表盘

用途：

- 汇总当天需要关注的事项
- 暴露最近变更与发布健康状况

最低能力：

- 草稿数量概览
- 最近发布内容概览
- 缺失必填字段提醒
- 陈旧草稿提醒
- 新建内容快捷入口

### GeekDaily

用途：

- 让期目创作、预览、发布足够快且不易误用

最低能力：

- 期目列表、编辑器、预览
- 按期号、日期、状态和关键词筛选
- 维护有序的期目条目列表
- 发布前校验 `episode_number` 唯一性
- 显示最终公共 URL

### 招聘

用途：

- 让工作人员发布结构化、可维护的岗位，而不是自由文本堆叠

最低能力：

- 列表、编辑器、已归档视图
- 按状态、地点、工作模式、远程支持和过期时间筛选
- 清晰显示岗位是否仍然在线
- 对缺失申请方式或已过期发布状态给出明确提示

### 文章

用途：

- 支撑长内容发布、预览和元数据维护

最低能力：

- 列表、编辑器、预览
- 按状态、作者、标签和关键词筛选
- 维护标题、摘要、正文、作者、标签、封面与 SEO 信息

### 活动

用途：

- 维护公开活动列表和详情，而不承担站内报名系统

最低能力：

- 当前活动、过往活动、编辑器
- 按状态、日期范围和城市筛选
- 区分即将开始与已结束活动
- 对时间范围和外部报名 URL 做显式校验

### 贡献者

用途：

- 维护贡献者资料和角色分组

最低能力：

- 贡献者列表
- 贡献者编辑器
- 角色管理
- 为贡献者绑定一个或多个角色
- 维护排序与公开状态

### 页面与导航

用途：

- 维护单例页面和全局站点配置

最低能力：

- 首页编辑
- About 编辑
- 页脚编辑
- 社交链接维护

设计要求：

- 用分区式表单，而不是裸 JSON 编辑
- 让单例页结构对运营人员可理解

### 媒体库

用途：

- 统一选择、复用和维护媒体资源

最低能力：

- 资源列表
- 上传流程
- 支持一次性多选并批量上传
- 删除未被内容引用的错误上传文件
- 资源详情视图
- 维护 alt text、用途、可见性和上传时间等 metadata

### 工作人员与权限

用途：

- 控制谁能访问 admin，以及他们能做什么

最低能力：

- 工作人员账号列表
- 角色列表与角色摘要
- 创建工作人员用户
- 分配角色
- 暂停或禁用访问

### 审计日志

用途：

- 让敏感运营变更可见、可追踪、可审查

最低能力：

- 查看操作人、动作、目标、时间戳
- 支持按操作人、模块、动作类型和时间范围筛选

## 共享 UX 规则

跨所有模块：

- 使用工作人员友好的语言，而不是原始 schema 术语
- 让发布状态足够醒目
- 校验提示必须具体且可执行
- 预览链接要靠近保存或发布操作
- 对重复项优先使用专门编辑器，而不是原始 JSON textarea
- 历史记录优先通过 archive 保留，而不是 hard delete

## 当前最低运营能力

工作人员应能在不接触数据库的前提下完成以下动作：

- 发布新的 GeekDaily 期目
- 发布并归档岗位
- 发布文章
- 发布带外部报名链接的活动
- 添加贡献者并分配角色
- 更新首页和页脚
- 上传并复用媒体

# 管理后台信息架构

## 目标

围绕工作人员任务来设计 Rebase admin，而不是围绕数据库 tables。

每个模块都应回答一个真实的运营问题，例如：

- 今天需要发布什么
- 哪一期 GeekDaily 正在编辑
- 哪些 jobs 当前在线
- 哪些 contributors 属于哪个角色

## 主要用户

- 超级管理员
- 内容编辑
- GeekDaily 编辑
- 招聘编辑
- 活动编辑
- 社区编辑
- 审计人员

## 顶层导航

推荐的顶层模块：

- 仪表盘
- GeekDaily
- 招聘
- 文章
- 活动
- 贡献者
- 页面与导航
- 媒体库
- 工作人员与权限
- 审计日志

## 仪表盘

目的：

- 展示今天需要关注的事项
- 暴露最近变更和发布健康状况

推荐组件：

- 按内容类型统计草稿数量
- 最新发布的 GeekDaily、article、job 和 event
- 长时间未更新的陈旧草稿
- 缺失必填字段的内容
- 系统健康概览
- 新建期目、新建 job、新建 article 和新建 event 的快捷操作

## GeekDaily 模块

目的：

- 让每日期目创作足够快，并且不容易误用

视图：

- 期目列表
- 新建期目编辑器
- 期目详情与预览

列表需求：

- 支持按期号、日期、状态和关键词筛选
- 清晰显示已发布与草稿
- 提供预览和编辑快捷链接

编辑器需求：

- 期号
- 标题，默认值为 `极客日报#{episode-number}`
- 摘要
- 导语 / 正文字段
- 发布时间
- tags
- 有序的推荐条目列表

期目条目编辑器需求：

- 条目标题
- 推荐人姓名
- 来源 URL
- 摘要
- 排序顺序

运营辅助能力：

- 复制上一期的结构
- 在发布前校验 `episode_number` 唯一性
- 显示最终公共 URL
- 为导语 / 正文提供 Markdown 预览
- 历史导入在 V1 中仍属于 bootstrap 或 seed 工作流，而不是单独的 admin 界面

## 招聘模块

目的：

- 帮助工作人员发布结构化、可读的 jobs，而不是倾倒原始文本

视图：

- job 列表
- job 编辑器
- 已归档 jobs 视图

列表需求：

- 支持按状态、地点、工作模式、是否支持远程和过期时间筛选
- 快速看出一个 job 是否仍然在线

编辑器需求：

- 公司或团队名称
- 岗位标题
- slug
- 薪资
- 是否支持远程
- 工作模式
- 地点
- 摘要
- 完整描述
- `apply_url`
- 备用联系方式标签 / 值
- 过期日期
- 状态

校验辅助：

- 要求至少提供 `apply_url` 或备用联系方式之一
- 当已发布 job 已过期时发出警告
- 展示最终公共详情 URL 以及 feed 中的展示预期

## 文章模块

目的：

- 支持更长篇的 Rebase 内容发布，并保持元数据与预览整洁

视图：

- 文章列表
- 文章编辑器
- 预览链接

列表需求：

- 支持按状态、作者、tag 和关键词筛选
- 显示发布时间、更新时间和 slug

编辑器需求：

- 标题
- slug
- 摘要
- Markdown 正文
- 作者
- tags
- 封面图
- SEO 标题和描述
- 状态

## 活动模块

目的：

- 干净地管理公开活动列表和详情页

视图：

- 当前活动
- 过往活动
- 活动编辑器

列表需求：

- 支持按状态、日期范围和城市筛选
- 对即将开始和已结束活动做出清晰视觉区分

编辑器需求：

- 标题
- slug
- 开始时间
- 结束时间
- 城市
- 地点
- 场地
- 摘要
- Markdown 正文
- 封面图
- 报名模式
- 外部报名 URL
- 状态

校验辅助：

- 开始时间必须早于结束时间
- 外部报名模式必须提供 URL
- event URL 预览应包含日期前缀

## 贡献者模块

目的：

- 管理人物与角色分组，同时避免让 contributor 维护退化成自由文本录入

视图：

- contributor 列表
- contributor 编辑器
- contributor 角色

Contributor 编辑器需求：

- 姓名
- slug
- 头像
- 标题语
- 简介
- 一个或多个角色
- 如 Twitter、WeChat、Telegram 等社交链接
- 排序顺序
- 状态

角色编辑器需求：

- 角色名称
- 角色 slug
- 描述
- 排序顺序

## 页面与导航模块

目的：

- 通过友好的分区式表单维护单例页面与全局站点配置

子分区：

- 首页
- About
- Footer
- 社交链接

首页编辑器应按分区组织，而不是暴露原始 JSON：

- 首屏区
- 当前信号
- 首页统计
- feed 引导区

About 编辑器应支持：

- 页面标题
- 页面摘要
- 有序的正文分区

Footer 编辑器应支持：

- 社交链接
- 支持项目
- 媒体资源
- 友情链接
- 版权信息

## 媒体库模块

目的：

- 集中管理可复用媒体的选择和 metadata 维护

视图：

- 资源列表
- 上传流程
- 资源详情抽屉或面板

推荐字段：

- 文件名
- mime type
- 字节大小
- 资源用途
- alt text
- 可见性
- 创建时间
- 上传所有者
- 如果未来需要，再加入使用引用

## 工作人员与权限模块

目的：

- 控制谁能访问 admin，以及他们能做什么

视图：

- 工作人员账号
- 角色
- 角色摘要

工作人员需求：

- 创建工作人员用户
- 分配一个或多个角色
- 暂停或禁用访问
- 在需要时留下备注

角色需求：

- 角色 code
- 名称
- 描述
- 绑定的权限
- 对关键角色的系统角色保护

## 审计日志模块

目的：

- 让运营变更可见、可审查

列表需求：

- 操作人
- 动作
- 目标类型
- 目标 id
- 时间戳
- 如果可用则显示 `request_id`

筛选应支持：

- 操作人
- 模块
- 动作类型
- 时间范围

## 共享 UX 规则

跨所有模块：

- 使用工作人员友好的语言标签
- 当原始 schema 细节无助于任务时隐藏它们
- 让发布状态足够醒目
- 让校验提示具体且可执行
- 把预览链接放在保存和发布控件附近
- 对重复项优先使用专门编辑器，而不是原始 JSON textarea
- 通过 archive 保留历史，而不是 hard delete

## 首个版本的工作流目标

工作人员应当能够在不接触数据库的情况下完成以下操作：

- 发布新的 GeekDaily 期目
- 发布一个新的 job，并在之后归档它
- 发布带封面图的 article
- 发布带外部报名链接的 event
- 添加一个 contributor 并分配一个或多个角色
- 更新首页和页脚
- 上传并复用媒体

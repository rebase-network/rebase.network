# 管理后台设计原则

适用于 `apps/admin` 中 Rebase 管理工作台的设计原则。

这个文件只回答三个问题：

- admin 要帮助工作人员完成什么
- 页面和模块应如何围绕任务组织
- 哪些交互与文案取向长期成立

颜色、字体、间距、组件和响应式硬规范见 `apps/admin/DESIGN.md`。

## 产品意图

管理后台是社区运营工作台，不是通用 CMS。

它应帮助工作人员：

- 高效发布和编辑内容
- 快速判断状态
- 在高频流程里避免犯错
- 以任务视角维护数据，而不是以 schema 视角操作系统

长期前提：

- 中文优先
- 桌面优先
- 运营效率高于展示感

## 核心原则

### 运营效率优先

- 每个页面的主任务必须立刻明确
- 高频操作应靠近它影响的内容
- 列表页应默认支持扫描、筛选和分页

### 主编辑区域必须占优

- 编辑页的主内容区必须保持视觉主导
- 次级元数据不应挤压正文编辑区域
- 低频设置应收进次级区块，而不是抢占首要位置

### 面向任务，而不是面向 schema

- 使用岗位、GeekDaily、文章、贡献者、媒体、工作人员这类任务词汇
- 不要让运营人员直接面对原始集合名、关系表思维或开发术语

### 能内联就内联

- 筛选器优先内联
- 相关控件尽量共享同一行
- 破坏性操作保持克制并远离主发布路径

## 页面类型优先级

### 列表页

- 长列表必须分页
- 搜索、数量和筛选应压缩在轻薄页头内
- 空列表仍要保留方向感和恢复路径

### 编辑页

- 标题、状态和主正文应尽早出现
- 不要在主编辑器之前堆叠低价值元数据
- 破坏性操作不能主导布局

### 设置与工具页

- 相关设置应按小面板分组
- 当两个工作流容易混淆时，用选项卡或分段模式

## 文案规则

- 默认使用简洁中文标签
- 删除重复常识的辅助文案
- 优先使用动作导向措辞
- 只有在工作流不直观时才补解释

好的例子：

- `开放岗位`
- `状态`
- `回到招聘列表`
- `新增工作人员`
- `上传文件`

## 反模式

默认不要引入这些模式：

- 大面积空白和过宽侧边栏
- 巨大的管理页首屏式页头
- 与任务无关的说明卡片
- 全宽破坏性按钮
- 明明可以内联却被拆散的筛选区
- 必须先滚动很久才看见表格的列表页

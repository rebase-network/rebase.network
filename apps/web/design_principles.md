# Web 设计原则

适用于 `apps/web` 中 Rebase 公共网站的设计原则。

这个文件说明公共产品意图与交互理念。
它不能替代 `apps/web/DESIGN.md`，后者保存的是硬性视觉规范。

## 产品意图

公共网站是一个社区媒体界面。

它应该帮助读者：

- 快速理解 Rebase 是什么
- 发现当前的社区动态
- 高效浏览归档内容
- 无障碍阅读长内容
- 在最少解释下切换到相关内容

它不应让人感觉像：

- 企业宣传册网站
- VC 作品集页面
- 通用内容农场
- 充满增长话术的极繁落地页

## 核心感受

公共网站应当给人的感觉是：

- 温暖
- 编辑感
- 活跃
- 有人味
- 有结构
- 值得信任

公共网站不应让人感觉：

- 冷
- 为了极简而极简
- 过度宣传化
- 相对于内容本身而言过度设计

## 核心原则

### 内容优先

- 阅读和浏览优先于装饰
- 主内容区块应立刻清晰可见
- 装饰性纹理应服务氛围，而不是争抢注意力

### 社区媒体高于营销

- 展示持续发生的社区活动迹象
- 让文章、GeekDaily、岗位和活动自己为社区发声
- 避免在同一页面的多个区块里重复相同定位语

### 紧凑，但不拥挤

- 归档页应紧凑且利于扫描
- 详情页应聚焦且易读
- 当下一个动作已经显而易见时，删除冗余分区文案

### 每个区块只承担一个明确目的

- 首屏区负责建立方向感
- 工具栏负责标识和导流
- 卡片负责总结并引导进入
- 侧边栏要么帮助行动，要么帮助理解

如果一个区块不能明确支持上述工作之一，就压缩它或删除它。

### 交互必须匹配可操作暗示

- 如果卡片在悬停时抬起，它通常应当可点击
- 如果一个动作是次要的，就把它做成次要样式
- 如果提供 RSS，就让它可见但低强调

## 页面类型指引

### 首页

- 可以是最具表现力的页面
- 应快速建立 Rebase 的气质与价值
- 应像社区的实时快照
- 不应变成一面由等权模块堆成的密集墙

### 归档页

示例：

- GeekDaily 列表
- 岗位列表
- 文章列表
- 活动列表

规则：

- 首屏区应简洁
- 支撑性区块应保持轻薄
- 列表内容应主导页面
- 当不需要完整分区标题时，使用工具栏

### 详情页

规则：

- 主内容正文应尽量靠上
- 除非会改变用户决策，否则不要在首屏区和正文里重复同一份元数据
- 相关内容区块应保持轻量
- 行动侧栏应紧凑且易读

## 文案指引

- 使用简洁、直接的标签
- 优先使用务实语言，而不是品牌口号式语言
- 避免那些只是重述明显页面结构的填充描述
- 保持分区标签足够短，能被快速扫描

好的例子：

- `开放岗位`
- `更多岗位`
- `回到招聘列表`
- `订阅 RSS`

避免：

- 已经可见列表上方的长篇归档说明
- 重复首屏区刚说过内容的标题
- 解释显而易见界面行为的辅助文案

## 公共交互规则

- 悬停意味着有动作
- 焦点必须保持可见
- 导航型卡片通常应是单击即可触发的目标
- 公共网站上很少有破坏性或高风险动作，不应在视觉上突出它们

## 响应式理念

- 公共页面必须同时适配桌面端和移动端
- 在桌面端，通过网格密度支撑扫描效率
- 在移动端，优先保住阅读清晰度，再考虑保留装饰性构图
- 先折叠布局复杂度，再降低文本可读性

## 反模式

默认不要把这些模式引入公共网站：

- 信息价值很低的巨大归档首屏区
- 对简单列表堆叠三层标题、眉标、副标题和辅助文案
- 不会改变用户选择的重复指标
- 面向公众内容里出现 CMS 式表单语言
- 几乎没有浏览价值的装饰性标签
- 不会通向任何地方的交互提示

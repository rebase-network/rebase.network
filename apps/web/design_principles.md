# Web 设计原则

适用于 `apps/web` 中 Rebase 公共网站的设计原则。

这个文件只回答三个问题：

- 公共网站要帮助读者完成什么
- 页面区块应如何分工
- 哪些交互与文案取向长期成立

颜色、字体、间距、组件和响应式硬规范见 `apps/web/DESIGN.md`。

## 产品意图

公共网站是一个社区媒体界面，不是宣传册站点。

它应帮助读者：

- 快速理解 Rebase 是什么
- 发现当前社区活动
- 高效浏览归档内容
- 顺畅阅读长内容
- 在相关内容之间自然跳转

## 核心原则

### 内容优先

- 阅读和浏览优先于装饰
- 主内容区块必须立刻清晰
- 装饰只服务氛围，不与内容争抢注意力

### 社区媒体高于营销

- 页面应展示持续发生的社区活动迹象
- 让文章、GeekDaily、岗位和活动本身承担内容表达
- 避免在同一页面重复同一套品牌定位语

### 每个区块只做一件事

- 首屏区负责建立方向感
- 工具栏负责标识和导流
- 卡片负责摘要和引导进入
- 侧边栏只在它确实帮助行动或理解时存在

### 交互必须诚实

- 看起来可点击的内容通常就应该可点击
- 次要动作必须保持次要样式
- RSS 应可见，但不应压过主要阅读路径

## 页面类型优先级

### 首页

- 用最少解释建立 Rebase 的气质和入口
- 应像社区活动快照，而不是等权模块堆叠

### 归档页

- 列表内容应主导页面
- 首屏区与辅助区块必须简洁
- 当完整分区标题没有增益时，使用工具栏

### 详情页

- 正文应尽早出现
- 不要在首屏区和正文里重复同一份元数据
- 相关内容区块保持轻量

## 文案规则

- 使用简洁、直接、可扫描的标签
- 优先使用务实语言，而不是口号式语言
- 删除那些只是在解释显而易见页面结构的文案

好的例子：

- `开放岗位`
- `更多岗位`
- `回到招聘列表`
- `订阅 RSS`

## 反模式

默认不要引入这些模式：

- 信息价值很低的巨大首屏区
- 简单列表上方的长篇解释
- 对用户决策没有影响的重复指标
- 面向公众页面里出现 CMS 式术语
- 没有去向的交互提示
